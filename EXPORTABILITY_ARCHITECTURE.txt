DENTVAULT — EXPORTABILITY & PORTABILITY ARCHITECTURE
=====================================================
Status: PROPOSAL — to be reviewed and planned before implementation
Intent: Discuss with Opus before coding anything


CORE INTENTION
--------------
The practitioner's entire working setup — patient data, settings, staff configuration,
document categories, dental chart tags, and all customisations — should live in a single,
self-contained folder (the Vault) that can be:

  • Backed up by copying one folder
  • Restored by pointing DentVault at a backup copy
  • Moved to a new machine without reconfiguration
  • Version-controlled or synced via any tool the user chooses (Time Machine, Dropbox, etc.)

Today the Vault folder holds patient files and documents, but the SQLite database
(which contains patient records, timeline entries, treatment plans, dental chart data,
and ALL settings) lives at a separate OS-level location determined by Tauri's app data
directory. This means the Vault folder alone is NOT a complete backup.


WHAT NEEDS TO MOVE INTO THE VAULT
----------------------------------
The SQLite database file (currently at Tauri's appData path, e.g.
  macOS: ~/Library/Application Support/com.dentvault.app/dentvault.db
  Windows: %APPDATA%\com.dentvault.app\dentvault.db
)
should be relocated into the Vault folder itself, e.g.:

  <vault_path>/dentvault.db

This single file contains:
  • All patients (demographics, medical history, etc.)
  • All timeline entries and treatment plans
  • All dental chart data (per-tooth conditions, surfaces, bridge groups)
  • All clinical exams
  • All documents metadata
  • Settings: dental chart tags, document categories, staff/roles, theme preference,
    staff members, any future user-configurable data


SETTINGS THAT ARE CURRENTLY STORED IN THE DB
---------------------------------------------
All of the following live in the `settings` key-value table or the `doctors` table:

  Key                  | Description
  ---------------------|------------------------------------------
  dental_tags          | Custom condition tags (label, color, pattern, shortcut)
  doc_categories       | Document folder categories (label, icon, key)
  staff_roles          | Staff role types (key, label, prefix)
  theme                | UI theme (light/dark)
  vault_path           | Path to vault itself (bootstrap-only, kept separately)

  doctors table        | All staff members (name, role, specialty, color)

These settings are entirely user-specific customisations. They should travel with the
database, not be left behind in the OS app-data directory when the vault is moved.


THE VAULT_PATH BOOTSTRAP PROBLEM
---------------------------------
There is a chicken-and-egg problem: the vault path itself must be stored somewhere
accessible BEFORE the vault is located. Currently vault_path lives in the settings table.
If the DB moves into the vault, we need a small separate bootstrap file to locate it.

Proposed solution:
  • Store ONLY the vault path in a tiny OS-level config file (Tauri's appData or
    similar) — e.g. ~/.dentvault_vault_path (a single line of text)
  • Everything else (all settings, all patient data) lives in <vault>/dentvault.db
  • On first launch, if no bootstrap file exists, prompt user to create or select vault
  • On subsequent launches, read the bootstrap file → open <vault>/dentvault.db

This keeps OS-level state minimal (just one path pointer) while making the vault
self-contained.


MIGRATION PATH FROM CURRENT ARCHITECTURE
------------------------------------------
Existing installations have the DB at Tauri's appData path. The migration flow:

  1. User opens Settings → "Move database to Vault"  (or prompted automatically)
  2. App copies dentvault.db from appData → <vault_path>/dentvault.db
  3. App writes the new bootstrap file pointing to vault_path
  4. App reopens the DB from its new location
  5. Old DB at appData location is left as a backup (not deleted automatically)

This migration should be offered once (non-destructive), not forced.


BACKUP & RESTORE UX
--------------------
Once the DB lives in the vault, the Settings page should expose:

  • "Open Vault Folder" — reveals the vault in Finder/Explorer so the user can
    manually copy it
  • "Create Backup" — copies the entire vault folder to a timestamped zip:
    e.g. DentVault_Backup_2026-03-14.zip
  • "Restore from Backup" — user selects a backup zip or folder; app validates it
    contains a dentvault.db, then switches to it (with confirmation)
  • "Change Vault Location" — existing feature, now also moves dentvault.db


MULTI-MACHINE / CLOUD SYNC CONSIDERATIONS
------------------------------------------
If the vault folder is synced via Dropbox / iCloud / OneDrive:
  • SQLite has known issues with concurrent access over network drives
  • DentVault is a single-user desktop app, so concurrent writes are not expected
  • We should document this limitation and recommend syncing only when the app is closed
  • Optionally: detect if the DB is on a network drive and warn the user

We should NOT try to build our own sync — just make the vault portable so the user
can use any sync tool they trust.


QUESTIONS TO RESOLVE WITH OPUS
--------------------------------
1. Tauri 2 API: What is the cleanest way to store a single bootstrap path outside the
   vault? (appLocalDataDir? a custom config file path? the Tauri store plugin?)

2. DB connection lifecycle: The current db.ts opens a connection once at app start.
   Moving the DB requires closing the old connection, copying the file, and reopening.
   What is the safest way to do this in a running Tauri app?

3. Should the DB migration be automatic on first launch (if vault is already set),
   or always opt-in via a Settings action?

4. Should we version the vault format? E.g. a vault_meta.json at the vault root that
   records the app version, DB schema version, and vault UUID — making it easier to
   validate backups before restoring.

5. File attachment handling: Documents are already stored in the vault folder tree.
   Once the DB is also in the vault, the entire vault is a single backup unit.
   Should we enforce that no absolute paths are stored in the DB — only vault-relative
   paths — so the vault remains portable if the vault folder itself is renamed/moved?
   (Check: does the current schema store absolute or relative document paths?)

6. What happens to the bootstrap file when the user changes vault location?
   It should be updated atomically (write new file → verify → delete old reference).


IMPLEMENTATION PHASES (rough sketch, pending Opus review)
-----------------------------------------------------------
Phase A — Bootstrap separation
  • Identify and isolate vault_path storage from the main settings table
  • Write vault_path to a separate OS-level tiny config (not the DB)
  • Ensure app can launch even if DB is not yet present at vault path

Phase B — DB relocation
  • Add "Move database to Vault" migration in Settings
  • Update db.ts to open DB from <vault>/dentvault.db when vault is configured
  • Handle connection lifecycle (close → copy → reopen)

Phase C — Backup/Restore UI
  • "Create Backup" (zip vault folder)
  • "Restore from Backup" (validate + switch)
  • "Open Vault Folder" shortcut

Phase D — Path portability audit
  • Audit all DB columns that store file paths
  • Normalise to vault-relative paths where possible
  • Ensure moving the vault folder doesn't break document links
